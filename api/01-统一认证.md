## 3. 接口本质说明

##### `accessibility` 和 `casp_ioc_domin` 是**同一套登录风控逻辑**中的两个不同参数，核心作用：**校验客户端是否为真实浏览器环境（而非自动化脚本/爬虫）**。

### 核心功能

- 校验客户端是否为真实浏览器环境，拦截自动化脚本、爬虫程序
- 生成携带**随机值 + 环境指纹**的校验参数，服务端依据该值判断是否放行登录请求
- 两个参数对应的接口在登录流程中**先后触发**，属于学校 CAS 系统典型反爬机制

---

### 参数说明

- **`accessibility`**：大概率与页面可访问性配置、环境检测逻辑相关
- **`casp_ioc_domin`**：系统内部配置项 / IOC 容器相关参数，用于风控校验

# 校园身份统一认证 - 登录接口

## 

## 1. 获取登录用户信息接口

### 1.1 基础信息

| 项目   | 说明                           |
|:----:|:----------------------------:|
| 接口名称 | 获取当前登录用户信息                   |
| 请求方式 | GET                          |
| 请求地址 | `/getLoginUser`              |
| 请求参数 | `t=随机时间戳`（防缓存）               |
| 接口用途 | 校验当前登录态、获取用户基础身份信息、区分访客/正式用户 |

### 1.2 请求头

```http
GET /getLoginUser?t=1715067891123 HTTP/1.1
Host: [school-domain]
Referer: https://[school-domain]/
User-Agent: [浏览器UA脱敏]
```

### 1.3 请求体

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "userId": "[学号/工号]",
    "userName": "[姓名]",
    "deptName": "[学院/部门]",
    "roleType": "student",
    "isLogin": true,
    "token": "[会话令牌]"
  }
}
```

### 1.4 技术分析

###### 从接口返回结果可以看出，系统在用户完成统一认证后，会通过 `/getLoginUser` 接口主动获取当前登录用户信息，用于前端初始化用户状态。

接口返回的数据中包含：

- **用户身份信息**
- **部门/学院信息**
- **登录状态**
- **会话 Token**
- **用户角色类型**

说明系统采用了典型的：

```text
登录态 + Token 会话校验
```

### 接口核心参数与业务逻辑分析

#### ✦ 请求参数说明

- **`t=随机时间戳`**
  用于**防止浏览器缓存接口数据**，确保每次请求都能获取到**最新的登录状态与用户信息**。

---

#### ✦ 关键字段与系统架构推测

从返回字段 `roleType`、`isLogin` 可分析出：

- 系统采用**角色权限体系**，不同角色展示不同页面与功能
- 前端**根据用户身份动态渲染页面**
- 登录状态由**后端统一校验**，安全性更高
- 接口具备**全局身份识别能力**

---

#### ✦ 接口调用时机（核心状态接口）

该接口为**登录态校验核心接口**，会在以下场景自动触发：

- 页面初始化加载
- 页面刷新（F5）
- 登录成功后
- 前端路由切换时
  作用：**自动确认用户是否已登录，并同步最新身份信息**。

## 3. 获取门户首页接口

### 3.1 基础信息

| 项目   | 说明                        |
|:----:|:-------------------------:|
| 接口名称 | 门户首页全局配置                  |
| 请求方式 | GET                       |
| 请求地址 | /getPortalConfig          |
| 请求参数 | 无                         |
| 接口用途 | 控制首页布局、主题风格、功能模块开关、卡片排序规则 |

### 3.2 请求头

```http
GET /getPortalConfig HTTP/1.1
Host: [school-domain]
```

### 3.3 请求体

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "themeMode": "light",
    "showBanner": true,
    "showSidebar": true,
    "cardList": ["PERSONALDATA","CALENDAR","NOTICE","SCORE"],
    "layoutType": "grid"
  }
}
```

### 3.4 技术分析

        从接口名称 `/getPortalConfig` 以及返回数据结构可以看出，该接口属于**门户首页初始化阶段的全局配置接口**，主要用于控制首页布局、主题风格以及功能模块显示逻辑。
接口返回了 `themeMode`、`showBanner`、`showSidebar`、`cardList`、`layoutType` 等关键字段，说明系统首页**并非固定的静态页面**，而是由后端动态下发配置数据后，再由前端完成页面渲染。
        其中 `cardList` 字段以数组形式定义了首页功能卡片的加载顺序与展示列表，表明系统采用了**组件化、模块化设计**，前端可根据配置动态生成首页内容。
结合此前分析到的 `vue.min.js`、`vue-router.min.js`、`axios.min.js` 等前端资源文件，可以进一步确定：
        该系统属于**典型的 Vue 单页应用（SPA）架构**，具备非常明显的**前后端分离**与**后端动态配置渲染**特征。

## 4.个人信息卡片接口 SYS_CARD_PERSONALDATA

### 4.1 基础信息

| 项目   | 说明                     |
|:----:|:----------------------:|
| 接口名称 | 个人信息卡片数据               |
| 请求方式 | GET                    |

### 4.2 请求头


### 4.3 请求体

```json
{
  "code": 200,

    "avatarUrl": "[脱敏头像地址]",

```

### 4.4 技术分析

        从接口名称 `SYS_CARD_PERSONALDATA` 及返回数据结构可判断，该接口为门户首页**个人信息专用卡片接口**，主要用于用户登录后，动态加载头像、学号、姓名、学院、专业、年级等基础身份资料。
        接口采用标准 JSON 格式返回数据，说明前端并非传统静态页面，而是通过 Vue 组件接收后端数据**动态渲染页面**，具备典型的前后端分离开发特征。
        请求携带 `t=时间戳` 参数，采用**防缓存机制**，避免浏览器缓存旧数据，保障首页展示的个人信息始终为最新状态。
        从接口命名 `SYS_CARD_` 前缀可进一步推测：系统首页采用**卡片化、模块化架构设计**，每一个功能卡片对应独立数据接口，由前端统一按需请求、拼装渲染至门户首页，是典型的智慧校园 SPA 门户建设方案。
